## 二、发布阻塞级别修复

### P0-1：知识 V3 / AI 聊天表未纳入备份

状态：确认需要修复。  
证据：

- `lib/core/services/backup_service.dart` 的 `_tableSpecs` 当前未包含：
  - `knowledgeSpacesV3`
  - `knowledgeMaterials`
  - `knowledgeCardsV3`
  - `knowledgeReviewLogsV3`
  - `tiantianQaSessions`
  - `tiantianQaMessages`
  - `aiChatMessages`
- `lib/core/database/app_database.dart` 和 `tables_extra.dart` 已注册这些表。

风险：

- 用户备份后恢复，会丢失知识空间、资料、卡片、复习记录、天天问答、AI 聊天历史。
- 恢复时如果旧 V3 数据残留，还可能出现新旧数据混合。

安全修复策略：

1. 只补备份/恢复，不改知识 V3 业务逻辑。
2. 给 7 张表添加 `_BackupTableSpec`，全部设为 `optional: true`，兼容旧备份。
3. 导入顺序必须遵守外键：
   - 先 `knowledgeSpacesV3`
   - 再 `knowledgeMaterials`
   - 再 `knowledgeCardsV3`
   - 再 `knowledgeReviewLogsV3`
   - 先 `tiantianQaSessions`
   - 再 `tiantianQaMessages`
   - `aiChatMessages` 独立处理
4. 恢复清理顺序必须反向，避免 FK 冲突。
5. 新增备份恢复测试：写入 V3 数据 -> 导出 -> 清库 -> 导入 -> 校验数量和关键字段。

验证命令：

```powershell

```

### P0-2：知识中文检索 bigram 为空字符串

状态：确认需要修复，但归入“知识卡片当前修复批次”。  
证据：

- `lib/features/study/utils/knowledge_tfidf_index.dart` 存在 `final bigram = '';`
- 当前测试覆盖了词典命中，如“机器学习”“操作系统”，但没有覆盖未知中文词的 bigram fallback。

风险：

- 非词典中文词检索失败。
- 用户导入资料后，搜索命中率看起来随机下降。

安全修复策略：

1. 把空 bigram 改为相邻两个未匹配中文字符组成的字符串。
2. 增加测试：
   - `TfidfIndex.tokenize('量子计算')` 至少包含 `量子`、`子计`、`计算` 中的合理 bigram。
   - 建索引后查询未知中文词可以命中对应文档。
3. 不重写整套 TF-IDF，不改旧知识 Repository。

验证命令：


## 三、高优先级但不应立刻大改

### P1-1：知识 V1/V2 与 V3 并行

状态：真实存在，但禁止本轮删除。  
证据：

- V3：`knowledge_v3_repository.dart`、`knowledge_v3_provider.dart`、`knowledge_workspace_page.dart`
- 旧版：`knowledge_card_repository.dart`、`knowledge_source_repository.dart`、`knowledge_card_provider.dart` 仍被 AI 上下文、旧测试、旧服务引用。

风险：

- 数据和逻辑有双轨。
- 贸然删除会破坏 AI 知识上下文、旧数据迁移、已有测试。

安全处理：

1. 本轮只做“路由入口统一到 V3”和“备份覆盖 V3”。
2. 不删旧表、不删旧 Repository、不 DROP 旧数据。
3. 等 V3 Workspace、导入、复习、AI 生成、备份恢复都稳定后，再做第二阶段迁移清理。
4. 第二阶段需要单独设计：
   - 老数据迁移完整性测试
   - 旧 Provider 引用清单
   - 旧路由全部重定向
   - 备份兼容旧表

### P1-2：V3 Repository 使用 raw SQL

状态：真实存在，但暂缓重构。  
原因：

- 当前知识卡片还在修复，raw SQL 虽不优雅，但不是立即数据丢失点。
- 直接重写成 Drift API 会扩大改动面，很容易引入新 bug。

安全处理：

1. 先修备份、检索和编译。
2. 只给 `_ensureTables()` 做一次性缓存/幂等优化可以考虑，但不是第一批。
3. Drift API 重构放到 V3 稳定后的专门重构批次。

### P1-3：喝水数据双轨残留

状态：部分已修，仍需后续收口。  
当前情况：

- 饮食页喝水按钮已改走 `waterPlanProvider.recordDrinkAmount()`。
- `waterPlanProvider` 会同步 `dailyWaterIntakeProvider`。
- 但 `settings_provider.dart` 里仍保留：
  - `dailyWaterIntakeProvider`
  - `todayWaterIntakeInitProvider`
  - `saveWaterIntake()`
- `diet_provider.dart` 仍从 `daily_water_intake` 读图表数据，这是可接受的读路径。

安全处理：

1. 不立刻删除 `dailyWaterIntakeProvider`，因为 Dashboard/统计可能仍依赖。
2. 先把所有写入口统一到 `waterPlanProvider`。
3. 给 `saveWaterIntake()` 标记弃用或改为转调 `waterPlanProvider`，避免未来新页面误用。
4. 保留 `daily_water_intake` 作为聚合缓存，`daily_water_records` 作为明细。是否合并数据结构，等稳定后再评估。

验证命令：

```powershell

flutter test test/features/health/water_plan_provider_test.dart
flutter test test/features/health/health_reminder_scheduler_test.dart
```

### P1-4：睡眠设置 key 命名不一致

状态：低风险真实问题。  
证据：

- `settings_page.dart` UI 定义里用 `daily_sleep_goal`
- 实际写入和读取用 `sleep_goal_hours`

判断：

- 当前功能不一定坏，因为保存时已经映射到 `sleep_goal_hours`。
- 但这是维护风险，后续容易误读 `daily_sleep_goal`。

安全处理：

1. 把设置页 GoalDefinition 的 key 统一为 `sleep_goal_hours`。
2. 保留兼容读取：如果老数据里存在 `daily_sleep_goal`，可迁移/兜底到 `sleep_goal_hours`。
3. 加一个 settings provider 测试。
